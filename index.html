Open-source WebRTC observability

rtcStats reads a WebRTC dump and tells you what went wrong on the call. AI reads the dump so you don't have to.

A client SDK traces every WebRTC call and streams it to a server, which stores it as a dump. Feature extraction reads that dump into a database you can query. You self-host all of it.

Try it

Quick start

The end-to-end example runs the whole pipeline locally: a minimal server, a browser window running WebRTC API calls, and a dump written to example/upload/ that you can import on rtcstats.com or the built-in dump importer.

git clone https://github.com/rtcstats/rtcstats
cd rtcstats
npm install
npm start --workspace=example
Reference: the WebRTC Stats Reference documents every getStats() metric rtcStats collects, its expected values, and common issues. AI agents such as Claude Code and Cursor can query analyzed sessions over the MCP server on the Enterprise plan.

The pipeline

A typical deployment

A full rtcStats deployment has a client SDK that collects data, a server that receives it as a dump, and an offline processor that extracts features. Storage and a database sit behind the server; the hosted service at rtcstats.com is an optional additional upload target.

  1. rtcstats-js (client-side)Wraps RTCPeerConnection and getUserMedia, traces every API call and event, and streams them to the server over a WebSocket.
  2. rtcstats-server (1..N)Receives the trace as a dump file, writes it to storage (S3 or compatible), and records metadata in Postgres. Stateless, so it scales horizontally against shared storage. Deploy it centrally; TLS is terminated by a load balancer in front.
  3. rtcstats-features (1..M)Polls the database for unprocessed dumps and extracts features offline. More CPU/memory intensive than collection, so it runs separately, in cheap regions or off-peak.
  4. rtcstats.com (optional)Configure the hosted service as an extra upload target to get Observations, an Experience Score, and a plain-English AI summary of what went wrong on the call.

       |   rtcstats-js   |
       |  (client-side)  |
       +--------+--------+
                |
                | Data Collection
                |
       +--------v----------+
       |  rtcstats-server  | (1..N)
       +-------------------+
                |
       +--------+----------+

+------v-------+    +------v--------+      +------------------+
|   Database   |    |    Storage    |<-----|  Visualization   |

How the pieces fit together

What the three packages share

Each stage reads what the previous one wrote. Four shared things make that work.

One dump file, three stages

rtcstats-js streams trace events over the WebSocket. rtcstats-server assembles them into a line-oriented JSON dump (the RTCStatsDump header and fileFormat version 3) and stores it; rtcstats-features reads that dump back to extract features. The file format is the contract between the server that writes it and the importers that read it.

SDK streams events → server writes the dump → features reads it

Identity flows end to end

Your app mints a JWT; rtcstats-js appends it to the WebSocket URL; rtcstats-server validates it and extracts user, session, and conference into the rtcstats-server database row that every extracted feature later hangs off of.

app JWT → SDK URL → server DB row → features

Compression is negotiated across the boundary

The client may apply method, stats-property, and getStats delta compression; importers (the server and features) must support all of it. So the client can send less data and the reader can always decode it.

SDK MAY compress · importer MUST decompress

Server and features share config and database

Configuration files in /config are shared between rtcstats-server and rtcstats-features, and features polls the same rtcstats-server table the server writes.

shared /config · shared Postgres
Client SDK

rtcstats-js

A JavaScript client SDK for monitoring WebRTC. It provides low-level logging of peerconnection API calls and periodic getStats() for analytics and debugging, inspired by Chrome's webrtc-internals and designed to add negligible overhead with minimal integration effort.

  • Transparent integration by overriding RTCPeerConnection et al. using adapter.js techniques
  • Traces every call and event on peerconnections and getUserMedia/getDisplayMedia
  • Streams over WebSocket to any rtcstats-server instance
  • Optional JWT auth via a rtcstats-token query parameter
  • Custom event tracing so app events land in the same dump
  • Reload counting to count page reloads during a call

Typical integration

import {wrapRTCStatsWithDefaultOptions} from '@rtcstats/rtcstats-js';

// Instantiate a trace function using the helper with default options.
const trace = wrapRTCStatsWithDefaultOptions();

// Connect to your rtcstats-server instance.
trace.connect('ws://localhost:8080' + window.location.pathname);

const pc = new RTCPeerConnection();

Under the hood the package exports WebSocketTrace (the WebSocket-backed trace function) plus wrapRTCPeerConnection, wrapGetUserMedia, and wrapEnumerateDevices for fine-grained wrapping.

Tracing your own events into the dump

Send app-specific events, such as an end-of-call user rating, so they sit alongside the WebRTC trace in one dump file. Custom method names must start with a colon and should not be compressed; the server does not process them by default, but importers still render them.

// method name (must start with ":"), peerconnection or null, data or null
trace(':userRating', null, {stars: 4});

For authenticating the connection with a JWT, see token authentication below, which covers the client and server halves together.

Collection server

rtcstats-server

The server listens on a WebSocket and writes the incoming data (the "dump") to a temporary file. If configured, the dump is uploaded to storage (S3 or compatible) and metadata is written to a database (Postgres or compatible). It does not handle TLS itself and assumes a load balancer terminates it.

  • WebSocket endpoint listening on port 8080 by default
  • node-config configuration shared with rtcstats-features under /config
  • S3-compatible storage plus Postgres metadata
  • JWT authorization extracting user, session, and conference
  • Docker-ready via the top-level Dockerfile and NODE_CONFIG
  • Optional rtcstats.com upload with sampling

Starting the server

npm start   # uses config/default.yaml, listens on :8080

Override without editing the defaults: add a gitignored development.yaml, select a production.yaml via NODE_ENV, or merge ad-hoc JSON through the NODE_CONFIG environment variable (the recommended approach inside Docker).

Dump file format

Dumps are line-oriented JSON: a RTCStatsDump header with a fileFormat integer, then one JSON array per event. See the dump file format spec for the element-by-element description, compression, and PII handling.

Forwarding to rtcstats.com

Alongside your own storage (not instead of it), the server can forward each dump to the hosted service for visualization and analysis. Disabled by default; set a token and, optionally, a randomPercentage in [0,1) to sample high-volume traffic.

Feature extraction

rtcstats-features

Feature extraction for peerconnection API traces and getStats() data. It polls the database for unprocessed dumps, downloads each one, extracts features, and writes them back, waiting five seconds between empty polls. It used to be part of the server; it now runs separately because it is more CPU and memory intensive than collection.

Four kinds of features

  • Server-side metadata, such as location
  • Client-wide, such as getUserMedia calls and errors
  • Per-RTCPeerConnection, from the API calls on each connection
  • Per-track, for each inbound/outbound track, mostly from getStats()

How the tables relate

Each extracted feature set hangs off the same rtcstats-server row the server wrote. A dump has one features_metadata, which has one features_client and many features_connection; each connection has many features_track.

  +-------------------+
  | rtcstats-server   |
  |-------------------|
+-+ id (PK)           |
| +-------------------+
| (1-to-1)
| +-------------------+
| | features_metadata |
| |-------------------|
| | id (PK)           +-+----------------------------------+
+-+ dump_id (FK)      | |                                  |
  +-------------------+ | (1-to-1)             (1-to-many) |
                        |                                  |
  +-------------------+ |          +---------------------+ |
  | features_client   | |          | features_connection | |
  |-------------------| |          |---------------------| |
  | id (PK)           | |        +-+ id (PK)             | |
  | dump_id (FK)      +-+        | | dump_id (FK)        +-+
  +-------------------+          | +---------------------+
                                 | +---------------------+
                                 | | features_track      |
                                 | |---------------------|
                                 | | id (PK)             |
                                 +-+ connection_id (FK)  |
                                   +---------------------+

Querying a dump's features

select * from "rtcstats-server" as server
    join features_metadata on features_metadata.dump_id = server.id
    join features_client on features_client.dump_id = features_metadata.id
    join features_connection on features_connection.dump_id = features_metadata.id
    join features_track on features_track.connection_id = features_connection.id
    where server.id = ${id}

Client and server, together

Token authentication, end to end

The client half and the server half of JWT auth only make sense together, and neither README can show the whole loop because each owns one end. A single shared secret ties your app, rtcstats-js, and rtcstats-server together, and the token carries the identifiers that end up on every dump.

  1. Your app signs a tokenYour app or signaling server signs a JWT with the shared secret and puts the identifiers under an rtcStats claim. For testing, bin/generate-token.js mints one from the configured secret.
  2. The SDK sends itrtcstats-js appends the token to the WebSocket URL as rtcstats-token when it connects. Nothing else on the client changes.
  3. The server verifies itrtcstats-server, configured with the same authorization.jwtSecret, verifies the token. A missing, invalid, or expired token closes the socket with a policy-violation (code 1008), which the client sees through its log callback.
  4. The identity lands on the dumpOn success the server reads rtcStats.user, rtcStats.session, and rtcStats.conference into the rtcstats_user, rtcstats_session, and rtcstats_conference columns of the dump's row, which is what features and analysis group and search by.
// 1. App / signaling server: sign with the shared secret.
//    The rtcStats claim carries the identifiers the server will store.
const token = jwt.sign({
  rtcStats: {
    user:       'user-123',    // long-lived user id     -> rtcstats_user
    session:    'session-abc', // short-lived session id -> rtcstats_session
    conference: 'room-42',     // groups a call's dumps  -> rtcstats_conference
  },
}, sharedSecret, {expiresIn: 600});

// 2. Browser (rtcstats-js): pass it on the WebSocket URL.
const trace = new WebSocketTrace({log: console.warn.bind(console)});
trace.connect('wss://example.com/?rtcstats-token=' + token);

// 3. Server (rtcstats-server config): the same secret validates it.
//    authorization:
//      jwtSecret: <sharedSecret>

No app wired up yet? Mint a token from the command line with the configured secret:

node bin/generate-token.js --lifetime 600
node bin/generate-token.js --url wss://example.com   # prints a full ws URL
The shared secret is the seam. authorization.jwtSecret on the server has to match whatever mints the tokens. That single value is why two packages, documented separately, actually interlock.

Rotating identity mid-page

There is one dump per WebSocket connection. To start a fresh session (a new session id, and a new dump row on the server) without reloading, close the connection and reconnect with a new token:

trace.close();
trace.connect('wss://example.com/?rtcstats-token=' + nextToken);

No cloud required

Inspect and anonymize dumps in the browser

If you are not using the hosted service at rtcstats.com, these browser-based tools work on dumps directly. Everything runs client-side, so nothing leaves your machine.

Dump importer

Open a Chrome webrtc-internals export or an rtcstats-js dump and explore it with zoomable graphs. Handy when you cannot use rtcstats.com.

Dump anonymizer

Strip IP addresses from a webrtc-internals or rtcstats dump before sharing it, so you can hand a dump to someone else without leaking network information.
